## 本轮实际流程

1. **查已知**：先读 [[concept.issue-native-agent-management]] 与 [[company.multica]]、[[company.raft]]、[[company.helio]]，带着“Issue 控制面 vs channel 协作 vs 预制员工”进入。
2. **官方面**：官网、README、Docs、npm、GitHub commits/releases、创始人 X 与视频，确认对象模型、时间线与类别语言。
3. **隔离体验**：在临时目录运行稳定版 onboarding，用无付费模型的 `process` dummy agent 检查数据库、公司、组织、Dashboard、成本、审批和活动面；没有把未运行模型冒充真实生产测试。
4. **规模面**：拆出精确月度流量、渠道、地域、品牌词、GitHub referral 与 YouTube social；没有把 Similar sites 自动列表当竞品。
5. **失败路径**：从 GitHub Issues 反查 budget、Hermes telemetry、heartbeat、retry、workspace、chat 与 secret；再用 LinuxDo 的真实体验校正官方叙事。
6. **团队与融资**：LinkedIn、GitHub 与 Designer Fund 交叉；融资没有强证据，因此不建 investment。
7. **资产化**：company、person、source、touchpoint、traffic、concept、note、图片与 X list 一并沉淀。

## 本轮有效的方法

- 对治理产品，**从 claim 反向寻找它依赖的 telemetry 与 enforcement contract**，比列功能更有价值。
- 隔离安装即使不调用模型，也能区分真实控制面和 landing-page shell；体验边界必须写清。
- Similarweb 的 Highcharts 原始 point 比页面摘要更稳定，可拿到月度 desktop/mobile 精确值；总和与 overview 校验后再写。
- GitHub Issue 不能按标题统计风险。优先选择有环境、版本、复现、代码或截图的报告，并保留 open/未复验边界。
- 中文转载适合判断类别语言扩散；LinuxDo/V2EX 更适合找真实 friction。两者不可混用。

## 需要改进

- Product Hunt 与 Reddit 的同名噪声仍高，需要更稳定的 entity-aware query；
- LinkedIn adapter 并发执行出现空输出，改成串行后只稳定抓到 Devin profile；团队履历仍有缺口；
- Similarweb 截图仍包含第三方界面 chrome，后续应优先做稳定 DOM 区块截图；
- 未做完整视频转录，只使用标题、描述、时间戳和可验证演示内容；若研究 founder origin，应补 transcript；
- 产品体验只覆盖 local single-player/no-model，下一轮同类产品应视风险决定是否增加真实 Agent 小任务与成本上限。

## 本轮对方法手册的增量

这轮与 Sapiom 的 control surface audit、Helio 的 trust contract audit、Multica 的 accepted output 判断相互印证，可以把“治理能力按控制环验证”升格为较稳定的方法；但具体检查项仍随产品类别变化，不固定成一套模板。
