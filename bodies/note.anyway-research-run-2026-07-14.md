# Anyway 本轮调研流程与反思

## 本轮路径

1. 查 vault，确认没有现成 Anyway 主体。
2. 官网/About/Docs 拆产品层，不把 homepage claim 直接当已实现能力。
3. npm/GitHub/CLI 复核真实发布节奏与命令面。
4. 对公开 marketplace directory 和 402 challenge 做无账号 smoke test。
5. 用 X 时间线重建 Beta、tracing、Workspace、SuperAPI、Wallet 的产品演化。
6. LinkedIn 与多源搜索核验创始人和团队，保留履历冲突。
7. Similarweb/Semrush 判断网站规模；遇 N/A 明确降级，不补数字。
8. HN/Reddit/PH/微信/小红书检索社区与中文世界，区分噪声、团队分发与独立反馈。
9. 提取官网原始产品图，不用整页截图替代产品素材。

## 本轮有效方法

- **时间线比静态功能表更能解释公司。** Anyway 五个月内四次扩边界，单看当前首页会误以为所有层同时成熟。
- **直接请求协议端点比只读文档多一层证据。** 不付款也能验证 directory 与 402 challenge 的真实结构。
- **目录字段也可暴露成熟度。** 37 个 service 全部 `verified=false` 是需要追问的产品边界。
- **网站流量和 API 使用必须分开。** Similarweb 无数据不能反证官方 wallet 数，也不能替官方数字背书。
- **Founder 履历要保留来源口径。** 公司 PR 与第三方数据库列出的经历不同，不应强行融合。

## 问题与下一轮改进

- LinkedIn employees 的 total 与可见列表差异大，需要更稳定的“当前全职”识别方法。
- Twitter list 命令本轮出现无输出，触点已写入 vault，但 list 加入需要复测。
- Product Hunt/HN/Reddit 的通用词噪声很高；搜索时应同时用域名、创始人、tagline 和产品名组合。
- Agent payment 类产品应增加固定的 policy audit：单笔/日/任务/供应商/类别、人类审批、退款、异常恢复、密钥边界。

本轮没有把新观察直接上升为通用 SOP；先作为 [[method.product-research-workflow-v0]] 的一轮样本保留，待与后续支付/治理产品比较。
